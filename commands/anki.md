# Anki デッキ生成パイプライン（最終版）

ユーザーから渡されたデータ・テーマ・教科をAnkiデッキ(.apkg)に変換する。
**全11フェーズを1つも省略せず実行**すること。

## 入力
$ARGUMENTS

---

## Phase 1: 入力分析 & 設計（4エージェント同時）

入力データを4つのサブエージェントで**同時に**分析する。

### Agent 1A — 教科・レベル・スコープ分析
```
【任務】入力データを分析し、以下を特定せよ。リサーチのみ、コードは書くな。

1. 教科/分野（例: 高校物理、TOEIC英単語、大学倫理学）
2. レベル（中学/高校/大学/資格試験/実務）
3. カバーすべき範囲を章立て・単元リストで整理
4. 入力データに含まれるキーワード・概念の総数を概算
5. この分野の学習者が最終的にどんな試験・場面で使うか推定

返却形式（JSON厳守）:
{
  "subject": "教科名",
  "level": "レベル",
  "scope": ["単元1", "単元2", ...],
  "keyword_count": 数値,
  "target_use": "想定される使用場面",
  "estimated_card_count": {"min": 数値, "target": 数値, "max": 数値}
}

入力データ:
（$ARGUMENTSの全文）
```

### Agent 1B — カードタイプ戦略設計
```
【任務】この分野に最適なカードタイプの使い分け戦略を設計せよ。リサーチのみ、コードは書くな。Web検索で「Anki 〇〇（分野名） おすすめ カード」等を検索し、実際の学習者の知見も取り入れよ。

各カードタイプの適用基準を設計:
- basic: どういう知識に使うか、front/backの書き方テンプレート
- reversed: どういう知識に使うか、例
- cloze: どういう知識に使うか、穴の開け方の基準
- typing: どういう知識に使うか

また、この分野特有の「こういう問い方をすると暗記しやすい」パターンがあればWeb検索で調べよ。

返却形式（JSON厳守）:
{
  "strategy": {
    "basic": {"use_for": "説明", "template": {"front": "テンプレ", "back": "テンプレ"}, "ratio": "全体の何%か"},
    "reversed": {"use_for": "説明", "template": {...}, "ratio": "..."},
    "cloze": {"use_for": "説明", "template": {...}, "ratio": "..."},
    "typing": {"use_for": "説明", "template": {...}, "ratio": "..."}
  },
  "field_tips": ["この分野特有のカード作成のコツ"]
}

入力データ:
（$ARGUMENTSの全文）
```

### Agent 1C — タグ体系 & デッキ構造設計
```
【任務】最適なタグ階層とデッキ構造を設計せよ。リサーチのみ、コードは書くな。

1. タグの階層構造を設計（例: 物理::力学::運動方程式）
2. サブデッキに分けるべきか、1デッキにまとめるべきか判断
3. 難易度タグ（★基礎 / ★★標準 / ★★★発展）の基準を定義
4. 横断的タグ（例: "計算あり", "図表あり", "頻出"）の設計

返却形式（JSON厳守）:
{
  "tag_hierarchy": {"教科": {"単元1": ["トピックA", "トピックB"], "単元2": [...]}},
  "deck_structure": "single|sub_decks",
  "deck_names": ["デッキ名1", ...],
  "difficulty_criteria": {"easy": "基準", "medium": "基準", "hard": "基準"},
  "cross_tags": ["タグ1", "タグ2"]
}

入力データ:
（$ARGUMENTSの全文）
```

### Agent 1D — 例文・具体例の収集戦略
```
【任務】この分野でカードに添えるべき例文・具体例のパターンを調査せよ。リサーチのみ、コードは書くな。Web検索必須。

1. この分野で暗記カードに例文を添える際のベストプラクティスをWeb検索で調べよ
2. 例文のフォーマット（長さ、形式、言語）の推奨を決めよ
3. 具体例の種類（数値例、日常例、歴史的エピソード、図解的説明）の使い分けを設計
4. 英語分野なら: 例文の難易度、ネイティブ表現の引用元
5. 理系分野なら: 典型的な計算例の提示方法
6. 文系分野なら: エピソード・逸話の活用方法

返却形式（JSON厳守）:
{
  "example_strategy": {
    "format": "例文のフォーマット説明",
    "length": "推奨文字数",
    "types": ["数値例", "日常例", ...],
    "per_card_guideline": "1カードにつき例文何個が最適か"
  },
  "sample_examples": [
    {"concept": "概念名", "good_example": "良い例文", "bad_example": "悪い例文", "why": "なぜ良い/悪いか"}
  ]
}

入力データ:
（$ARGUMENTSの全文）
```

**✅ Phase 1 完了時**: 4エージェントの結果を統合し、設計方針サマリーを表示。

## Phase 2: 並列ディープリサーチ & カード生成（4エージェント同時）

Phase 1 の設計方針を踏まえ、4エージェントで**同時に**カードを生成する。

### Agent 2A — 基礎概念カード生成
```
【任務】入力データの基礎概念・定義・用語を全てカード化せよ。
リサーチのみ、コードは書くな。Web検索を必ず使い、正確な定義を取得せよ。

対象:
- 全ての用語の定義（正式な定義 + 噛み砕いた説明）
- 公式・法則（導出過程ではなく、覚えるべき形）
- 人名・年号（関連する業績とセットで）
- 基本的な分類・カテゴリ

各カードのルール:
- front は簡潔に。1つの問いに1つの答え（最小情報原則）。
- back には答え + 補足説明 + 例文/具体例を必ず含めよ。
- 例文は「なるほど、そういうことか」と腹落ちするものを選べ。
- cloze は文脈から答えが推測できる自然な文にせよ。
- **最低25枚以上**生成せよ。

【例文の付け方の例】
- 英単語の場合（品詞は日本語、発音記号は不要、語源を必ず含めること）:
  front: "<b>ubiquitous</b><br><span style='color:#666; font-size:14px;'>形容詞</span>"
  back: "どこにも存在する、遍在する<br><br><i>語源: ラテン語 ubique（どこでも）= ubi（どこ）+ -que（〜でも）</i><br><br><i>例: Smartphones have become <b>ubiquitous</b> in modern society.</i><br><i>（スマートフォンは現代社会で遍在するようになった）</i>"
  word: "ubiquitous"  ← TTS用（plain text、tts=True時に必須）

  品詞は日本語で表記: 名詞, 動詞, 形容詞, 副詞, 前置詞, 接続詞, 代名詞, 間投詞
  複数品詞がある場合: "名詞/動詞" のように併記
  発音記号（IPA）は不要。ユーザーはフォニックスで読めるので一切付けないこと。
  語源は必須: ラテン語・ギリシャ語の語根、接頭辞・接尾辞の分解、意味の由来を記載

- 物理の場合:
  front: "運動エネルギーの公式は？"
  back: "<b>K = ½mv²</b><br><br>m: 質量 [kg]、v: 速度 [m/s]<br><br><i>例: 2kgの物体が3m/sで動く → K = ½×2×3² = 9 J</i>"

- 倫理の場合:
  type: cloze
  text: "{{c1::カント}}は{{c2::『純粋理性批判』}}({{c3::1781}}年)で、人間の認識能力の限界を論じた"
  extra: "批判哲学の出発点。「理性の法廷」に全てをかける姿勢。<br><i>覚え方: 「いーなはい(1781)、カントの批判」</i>"

- 歴史の場合:
  front: "ウェストファリア条約（1648年）の意義は？"
  back: "<b>主権国家体制の確立</b><br><br>・三十年戦争の講和条約<br>・各領邦の主権を承認<br>・「国際法の出発点」とも<br><br><i>例: 現代の「内政不干渉原則」の源流。国連憲章2条7項にもこの思想が反映されている。</i>"

返却形式（JSON厳守）:
{
  "agent": "2A",
  "cards": [
    {
      "type": "basic|reversed|cloze|typing|listening",
      "front": "...", "back": "...",
      "text": "...(cloze用)", "extra": "...(cloze用)",
      "word": "TTS用plain text（英語カードの場合必須）",
      "pos": "品詞を日本語で（英単語の場合必須。例: 名詞, 動詞, 形容詞, 副詞）",
      "etymology": "語源（英単語の場合必須。語根・接頭辞・接尾辞の分解と由来）",
      "tags": ["教科", "単元", "トピック"],
      "source": "情報源URL or 'common knowledge'",
      "confidence": "high|medium|low",
      "difficulty": "easy|medium|hard",
      "example_included": true/false,
      "why_this_card": "このカードが暗記に必要な理由"
    }
  ],
  "coverage_gaps": ["カードにできなかった概念"]
}

入力データ:
（$ARGUMENTSの全文）

設計方針:
（Phase 1の統合結果をここに貼る）
```

### Agent 2B — 比較・対比・誤解カード生成
```
【任務】「違い」「比較」「よくある間違い」に特化したカードを生成せよ。
リサーチのみ、コードは書くな。Web検索を必ず使え。

対象:
- 混同しやすい概念ペアの比較（「AとBの違いは？」）
- よくある誤解・間違い（「〇〇は△△だと思われがちだが、実は…」）
- 似た用語の使い分け
- 対立する理論・学説の比較

各カードのルール:
- 比較カードの back には必ず表形式 or 箇条書きで差異を明示
- 「なぜ混同されるのか」の説明を添えよ
- 具体例で違いを示せ
- **最低20枚以上**生成せよ

【比較カードの例】
- front: "速さ (speed) と 速度 (velocity) の違いは？"
  back: "<table><tr><th></th><th>速さ</th><th>速度</th></tr><tr><td>種類</td><td>スカラー量</td><td>ベクトル量</td></tr><tr><td>方向</td><td>なし</td><td>あり</td></tr><tr><td>負の値</td><td>取らない</td><td>取りうる</td></tr></table><br><i>例: 車が時速60kmで北に走る → 速さ=60km/h、速度=60km/h 北向き</i>"

- type: cloze
  text: "帰納法は{{c1::個別の事例から一般法則を導く}}推論であり、演繹法は{{c2::一般法則から個別の結論を導く}}推論である"
  extra: "<i>帰納の例: 「このカラスは黒い、あのカラスも黒い → 全てのカラスは黒い」<br>演繹の例: 「全ての人間は死ぬ、ソクラテスは人間だ → ソクラテスは死ぬ」</i>"

返却形式: （2Aと同じJSON形式）

入力データ: （$ARGUMENTS全文）
設計方針: （Phase 1結果）
```

### Agent 2C — 体系・因果・流れカード生成
```
【任務】構造的理解・因果関係・時系列の流れに関するカードを生成せよ。
リサーチのみ、コードは書くな。Web検索を必ず使え。

対象:
- 分類体系（「〇〇を3つに分類せよ」）
- 因果関係（「AがBを引き起こす理由は？」）
- 時系列の流れ（歴史の場合: 出来事の順序、科学の場合: 発展の経緯）
- 前提→結論の論理構造
- 上位概念→下位概念の包含関係

各カードのルール:
- 列挙系は覚えやすい語呂合わせ・頭文字暗記法も添えよ（Web検索で調べよ）
- 因果カードは「なぜ?」を front にして具体的メカニズムを back に
- 流れ系は cloze で順序を穴埋めにせよ
- **最低20枚以上**生成せよ

【体系カードの例】
- front: "マズローの欲求5段階を下から順に挙げよ"
  back: "1. <b>生理的欲求</b>（食事、睡眠）<br>2. <b>安全の欲求</b>（住居、健康）<br>3. <b>社会的欲求</b>（所属、愛情）<br>4. <b>承認の欲求</b>（尊敬、地位）<br>5. <b>自己実現の欲求</b>（成長、創造）<br><br><i>覚え方: 「生(せい)・安(あん)・社(しゃ)・承(しょう)・自(じ)」= せいあんしゃしょうじ</i>"

- type: cloze
  text: "三権分立: {{c1::立法権}}は国会、{{c2::行政権}}は内閣、{{c3::司法権}}は裁判所が担う"
  extra: "モンテスキュー『法の精神』(1748年)で提唱<br><i>相互に抑制と均衡(checks and balances)を図る仕組み</i>"

返却形式: （2Aと同じJSON形式）

入力データ: （$ARGUMENTS全文）
設計方針: （Phase 1結果）
```

### Agent 2D — 試験対策・計算・実践カード生成
```
【任務】試験頻出ポイント・計算パターン・実践的知識のカードを生成せよ。
リサーチのみ、コードは書くな。Web検索を必ず使え。
「〇〇 試験 頻出」「〇〇 テスト よく出る」「〇〇 過去問 傾向」で検索せよ。

対象:
- 試験で頻出の知識（Web検索で過去問傾向を調べよ）
- 引っかけ問題のパターン（「正しいのはどれか」系）
- 典型的な計算問題と解法手順
- 実務・日常で使える実践的知識
- 覚え方のコツ・語呂合わせ（Web検索で調べよ）

各カードのルール:
- 計算系は具体的な数値例を必ず含めよ
- 引っかけパターンは「なぜ引っかかるのか」を解説
- 語呂合わせがWeb上にあれば積極的に採用
- typing カードを積極的に使え（正確な用語・スペルを打たせる）
- **最低20枚以上**生成せよ

【試験対策カードの例】
- front: "等加速度直線運動の3公式を全て書け"
  back: "① <b>v = v₀ + at</b><br>② <b>x = v₀t + ½at²</b><br>③ <b>v² = v₀² + 2ax</b><br><br><i>覚え方: 「ブイブイ関係(①)、距離は時間の2次式(②)、時間消去(③)」</i><br><i>例: v₀=0, a=2m/s², t=3s → v=6m/s, x=9m</i>"

- type: typing
  front: "「遍在する」を英語1語で（u_______）"
  back: "ubiquitous"

- front: "【引っかけ注意】慣性の法則で、等速直線運動する物体に働く力は？"
  back: "<b>合力 = 0（ゼロ）</b><br><br>⚠️ 「力が働いていない」ではない！<br>力は働いているが、合力がゼロ。<br><br><i>例: 机の上の本 → 重力↓と垂直抗力↑が釣り合って合力0</i>"

返却形式: （2Aと同じJSON形式）

入力データ: （$ARGUMENTS全文）
設計方針: （Phase 1結果）
```

**✅ Phase 2 完了時**: 4エージェントの結果を集約。総カード数と内訳を報告。

## Phase 3: 例文強化 & 補足情報追加（4エージェント同時）

Phase 2 で生成された全カードを4等分し、各エージェントが担当範囲の**全カードに例文・具体例・補足情報を追加・強化**する。

### 全エージェント共通指示:
```
【任務】担当範囲の全カードについて、例文・具体例・補足情報を強化せよ。
リサーチのみ、コードは書くな。Web検索を必ず使え。

以下の基準で各カードを評価し、不足があれば強化:

1. 例文チェック:
   - example_included: false のカードには必ず例文を追加
   - 既存の例文が抽象的すぎる場合はより具体的なものに差し替え
   - 英語系: 実際の用例をWeb検索（例文検索サイト、ニュース記事等）
   - 理系: 具体的な数値を使った計算例
   - 文系: 歴史的エピソード、現代の具体事例

2. 補足情報チェック:
   - 語源情報があると暗記に役立つ場合は追加（特に英単語）
   - 語呂合わせ・覚え方がWeb上にあれば追加
   - 関連する豆知識で記憶の定着を助けるものがあれば追加
   - cloze カードの extra が空欄の場合は必ず補足を追加

3. フォーマット統一:
   - 例文は <i>例: ...</i> のHTML形式で統一
   - 覚え方は <i>覚え方: ...</i> で統一
   - 語源は <i>語源: ...</i> で統一
   - 太字 <b>、改行 <br> を適切に使って視認性を高めよ

改善したカードのみ返却。変更なしのカードは返さなくてよい。

返却形式（JSON厳守）:
{
  "enhancer": "3A|3B|3C|3D",
  "range": "カードX〜Y番",
  "enhanced_cards": [
    {"card_index": 番号, "enhanced_card": {改善後のカード全体}}
  ],
  "still_missing_examples": [例文を見つけられなかったカード番号]
}

担当カード:
（担当範囲のカードデータをJSON配列で渡す）
```

**✅ Phase 3 完了時**: 例文追加数、補足追加数を報告。

---

## Phase 4: 統合 & 正規化

Phase 2-3 の結果を**自分で**統合する:

1. Phase 3 の強化結果を Phase 2 のカードに反映
2. 重複排除（front/text が実質同じカードを検出し、質の高い方を残す）
3. タグ正規化（Phase 1C のタグ体系に統一）
4. カードタイプ最適化（Phase 1B の戦略に基づく）
5. 難易度バランス確認（easy/medium/hard の分布）
6. 全カードに通し番号を振る
7. coverage_gaps（Phase 2 で報告されたもの）を一覧化

統合後のカード総数・タイプ別内訳・タグ別内訳を報告。

---

## Phase 5: 並列4観点クロスチェック（4エージェント同時）

Phase 4 の統合済み全カードを4つのチェッカーで**同時に**検証。

### Checker 5A — ファクトチェック（正確性の鬼）
```
【任務】全カードの事実関係を検証せよ。リサーチのみ、コードは書くな。

- 全カードに目を通し、怪しいものはWeb検索で裏取り。
- confidence: "medium"/"low" は全件Web検索で確認。
- 数値（年号、物理定数、統計値、人口、距離等）は必ずWeb検索。
- 例文の内容が事実として正しいかも確認（架空の例なら問題ないが、事実として述べている例は裏取り）。
- 間違いには修正版を提示。確認できないものは "unverified"。

返却形式（JSON厳守）:
{
  "checker": "fact_check",
  "total_checked": 数値,
  "errors_found": [{"card_index": 番号, "problem": "誤り", "correct_info": "正解", "source": "URL"}],
  "unverified": [番号リスト],
  "examples_with_errors": [{"card_index": 番号, "example_problem": "例文の誤り", "fix": "修正"}],
  "verdict": "品質評価"
}

全カードデータ:
（Phase 4の統合済み全カードJSON）
```

### Checker 5B — 網羅性チェック（抜け漏れの鬼）
```
【任務】重要事項が漏れていないか検証せよ。リサーチのみ、コードは書くな。

- Web検索で教科書の目次・シラバス・試験範囲を調べ、カバー率を評価。
- 漏れている重要概念は追加カードを作成（例文・具体例付きで）。
- Phase 2 の coverage_gaps がカバーされているか確認。
```

## 鉄の掟（違反は最初からやり直し）

1. **Phase 1, 2, 3, 5, 7, 8 では必ず4つのAgentを1メッセージで同時に呼べ**（逐次起動禁止）
2. **サブエージェントには「コードを書くな、リサーチ/チェックのみ」と必ず明示**
3. **サブエージェントには「Web検索を必ず使え」と必ず明示**
4. **全サブエージェントにJSON形式での返却を強制**
5. **Phase を1つでも飛ばすな。「問題なかったので省略」も禁止**
6. **confidence: low のカードは Phase 9 で必ず削除**
7. **カード最低30枚**（足りなければ Phase 2 を再実行）
8. **数値は必ずWeb検索で裏取り**
9. **全カードに例文 or 具体例 or 覚え方のいずれかを含めよ。裸のカード禁止**
10. **各Phase完了時に「✅ Phase X 完了」と報告**
11. **例文は <i>例: ...</i> 、覚え方は <i>覚え方: ...</i> のHTML形式で統一**
12. **サブエージェント合計: 最低24体使用すること（6フェーズ × 4体）**
13. **英語関連デッキでは必ず tts=True を指定し、全カードに "word" フィールド（plain text）を含めよ**
14. **英語デッキでは listening カードを最低5枚含めよ（リスニング力強化）**
15. **英単語カードには必ず品詞（日本語: "名詞", "動詞", "形容詞" 等）と語源を含めよ。品詞・語源なしの英単語カード禁止**
16. **英単語の front には品詞を `<span style='color:#666; font-size:14px;'>形容詞</span>` 形式で表示。発音記号(IPA)は一切付けるな**
17. **英単語の back には語源を `<i>語源: ...</i>` 形式で必ず含めよ（ラテン語/ギリシャ語の語根分解）**
